User-facing index: [../../product/requirements.md](../../product/requirements.md). The AIGM behaviour contract is [../aigm.md](../aigm.md).

# Content Generation — Detailed Spec

Authoring, generating, persisting, and surfacing game content (characters, NPCs, monsters, maps, tokens, encounters, adventures), the UI panels and overlays that present it, the AIGM that improvises around it, and the REST API that serves it.


- **US-002** As a player, I want to choose a character from the available roster before the encounter begins so that I can experience different combat styles. Each character is one JSON file in `server/data/characters/` (`PlayerDef`) declaring its identity (name, species id, lineage, background id, feat ids), level, ability scores, skill proficiencies, saving-throw proficiencies, `defaultEquipment`, and — for spellcasters — `spellcastingAbility`, `defaultCantripIds`, `defaultSpellbookIds`, `defaultPreparedSpellIds`, `defaultSpellSlots`. The server reads them via `GET /characters`; the Encounter Setup Scene renders one card per file. Adding a new character is a JSON drop with no code change. The shipped roster currently includes a Fighter, a Rogue, and a Wizard example to exercise the three combat archetypes (melee, ranged, caster).
- **US-003** As a player, I want to see my character's stats in a toggleable sidebar so that I can inspect my current HP, AC, speed, ability scores, and XP without interrupting play. The Player Panel displays name, species/class/level, a colour-coded HP bar, combat stats (AC, Speed, Proficiency bonus, Initiative), all six ability scores with modifiers, current XP, quests, context-sensitive combat action buttons (ATTACK, CAST, THROW…, Dash, Dodge, Disengage, Second Wind, Hide, RELEASE [Concentration], End Turn, Roll Death Save), exploration-phase buttons (★ LEVEL UP when XP qualifies, ☾ LONG REST when the encounter permits it), and the persistent buttons CHARACTER (opens the Character Sheet — see US-038), TALK, and SEARCH. The panel is hidden by default and toggled open/closed by clicking the player token on the Game Map. The ATTACK button is always visible during the exploring and player_turn phases: it appears green and enabled when a living non-ally NPC is selected and within melee range, and grey and disabled otherwise.
- **US-004** As a player, I want to select an AI Dialogue encounter so that I can have a freeform conversation with an AI-powered NPC. Superseded by US-021; the `ai_dialogue` encounter type has been removed and merged into `social_interaction`. All NPC encounters now use AI dialogue by default with riddle fallback.
- **US-006** As a player, I want to pick up, manage, and use items in an inventory so that I can equip myself for challenges. Up to 3 Health Potions (2d4+2 healing per SRD) spawn as green diamond tokens on the map each encounter; walking onto a token picks it up automatically. XP is recorded in the event log on each kill. Gold is not automatically awarded — it must be given by the AIGM via the `award_gold` tool (e.g. as quest rewards or loot narrated in the GM overlay). Current GP is shown in the Inventory Overlay stats bar.
- **US-007** As a player, I want to receive and track quests so that I have clear goals and a sense of progression. Each encounter declares a one-line player-facing **objective** in its JSON (`EncounterDef.objective`); when omitted, a default is derived from the encounter's `encounterTypes` ("Defeat the hostile creatures", "Speak with the locals and resolve the situation", "Search the area for hidden secrets"). The objective is surfaced on `GameState.objective` and rendered as the **OBJECTIVE** row at the top of the Quests section of the Player Panel — accent-coloured so it reads as the player's overarching goal. Quests are sub-objectives assigned automatically at encounter start based on the active encounter types. Simple Combat assigns: First Blood (slay 1 enemy, +10 XP +5 GP), Slay All (slay every enemy spawned, +25 XP +15 GP), and Treasure Hunt (collect 2 Health Potions, +10 XP +5 GP). Exploration assigns: Keen Eye (find 2 secrets, +15 XP +10 GP). Social Interaction assigns: Make Contact (open a conversation with any NPC, +10 XP +5 GP, awarded on the first AI reply or riddle answer). The QUESTS row beneath OBJECTIVE lists each quest as "· Title N/M" while in progress and "✓ Title" when complete. Quest rewards are added to XP and GP and logged in the Event Log on completion.
- **US-008** As a player, I want my game state saved and loaded automatically so that I can continue where I left off. Save state is split into two files on the server: a **character save** at `server/data/saves/{characterId}.json` (HP, XP, gold, inventory, equipment, spell slots, prepared spells, the generic `resources` map that holds class-feature usage counters such as Second Wind — see US-066 — plus the level-up runtime fields `tracks`, `subclassId`, `arcaneRecoveryUsed`, `sneakAttackUsedThisTurn`, `pactMagic`, `mysticArcanum`, and the runtime `mageArmor` / `concentratingOn` flags — see US-091 and US-065) and a **world save** at `server/data/saves/world.json` (full session/encounter state, referencing the character by `defId`). On boot, `BootScene` calls `GET /world`; if a world save exists the session is restored and `GameScene` opens directly, otherwise `EncounterSetupScene` is shown. When the player reloads the browser and reconnects to an active session, the Introduction Overlay is suppressed — play resumes directly from the saved state. The character save is written on every action and after every AIGM exchange. The world save is deleted when the player presses NEW ENCOUNTER. Each character card in the setup screen shows HP / XP / GP and equipped gear from the character save; a DELETE SAVE button clears the character save without leaving the setup screen.
- **US-016** As a player, I want to click a creature in the game map to select it so that I can inspect its stats. The selected creature is highlighted with a coloured outline. A Target Panel appears on the right side of the Game Map showing the creature's name, HP bar, AC, speed, and ability scores — mirroring the layout of the Player Panel. Clicking elsewhere or defeating the creature deselects it and hides the panel. When combat starts, the nearest active enemy is automatically selected.
- **US-018** As a player, I want to zoom in and out on the game map using the mouse wheel so that I can get a better view of the battlefield. I also want to pan around the map by left-click dragging.
- **US-020** As a player, I want to be presented with an encounter setup menu before the game starts so that I can choose my character and encounter. Superseded by US-029; the setup scene now shows a two-column layout (Character | Premade Encounter) rather than the original three-column free-form selector.
- **US-021** As a player, I want to select a Social Interaction encounter so that I can engage with an NPC rather than fight. All creature communication — with NPCs and enemies alike — goes through the Game Master overlay (Claude Sonnet). The player opens the GM overlay and addresses any creature by name or pronoun; the GM responds in that creature's voice, constrained by its persona and knowledge. Enemies may taunt, plead, or bargain. The GM calls `complete_quest('make_contact')` automatically when the player first successfully engages an NPC in a social encounter.
- **US-022** As a player, I want to select the Exploration encounter type so that I can discover hidden secrets on the map. When active, 4 secrets are placed on the map (invisible to the player). The **SEARCH** action is now a permanent **green Action button** in the Player Panel's action stack (above MOVE), available in both `exploring` (free, no economy) and `player_turn` (costs the full Action — no Cunning Action fast-track per SRD). One Wisdom (Perception) roll resolves three things at once on success vs the relevant DC: (a) **secrets** on adjacent tiles (gold / Health Potion / lore inscription, DC 10/12/15); (b) **corpses** on adjacent tiles with an authored `corpseSearch` payload — one-shot, the engine flips the corpse to `SEARCHED` so the AIGM does not roll a second Perception check on the same body (see `set_npc_dead` in US-068 for authoring); (c) **hidden NPCs** within 6 tiles via `runPerceptionSweep`. Each secret and corpse payload resolves only once. **SRD scope** (`Search_[Action].md`): the SRD Search action is a Wisdom check with **four** applicable skills depending on what's being detected — **Perception** (concealed creature or object), **Insight** (creature's state of mind), **Medicine** (creature's ailment or cause of death), and **Survival** (tracks or food). The current implementation only resolves Perception; the Insight / Medicine / Survival variants run through the AIGM's `request_ability_check` flow.
- **US-023** As a player, I want to play on hand-crafted saved maps so that I can experience specific layouts. Maps are JSON files in `server/data/maps/` served via `GET /maps`. Map files are stored in **Tiled-compatible JSON** (palette + integer GIDs, see [data-model.md](../data-model.md)) and are pure geometry — tile semantics (`passable`, future difficult/cover/traps) come from a three-tier lookup at session-creation time: the encounter's `tileProperties` overrides win first, then the tileset legend (`server/data/tilesets/*_legend.json`) supplies defaults, then any unknown GID falls back to impassable. The same map can be reused across encounters with different mechanics by overriding only the GIDs that differ. The combined `passable` grid is built per-session by `SessionBuilder.buildGameMapFromSaved`. Map selection is implicit in the encounter choice (each `EncounterDef` carries a `mapId`); the standalone map picker overlay is no longer used. Adding a map is a JSON drop with no code change; the shipped set is content, not a requirement.
- **US-026** As a developer, I want all game content served from the Fastify REST API so that content can be updated centrally without redeploying the client. Static JSON files live under `server/data/` in per-type subdirectories. The server exposes `GET /characters`, `GET /monsters`, `GET /npcs`, `GET /factions`, `GET /conversations`, `GET /equipment`, `GET /feats`, `GET /backgrounds`, `GET /species`, `GET /spells`, `GET /features`, `GET /classes`, `GET /subclasses`, `GET /maps`, and `GET /encounters`; each endpoint reads all `.json` files in its directory and returns them as an array. The client preloads all collections in `BootScene` and stores them in the Phaser registry.
- **US-027** As a developer, I want NPCs to be a distinct data type from monsters so that non-combat characters can carry dialogue personas without polluting SRD stat blocks. NPCs live in `server/data/npcs/` as `NPCDef` objects with `id`, `name`, `monsterClass` (reference to a `MonsterDef` id for stat resolution), `color`, and an optional `persona` string. At spawn time the client merges the NPC's identity over the referenced monster's stat block. The `POST /npc/chat` endpoint looks up the NPC by id, resolves its persona, and forwards the conversation to the Claude API. Adding an NPC is a JSON drop with no code change; the shipped roster is content, not a requirement.
- **US-028** As a player, I get an introduction to the encounter when I start it so that I feel more immersed in it. When the game map loads, an Introduction Overlay appears before the player can act. It shows the active encounter type chips, the player's name and class, and a narrative introduction paragraph. The overlay must be dismissed before play begins. The encounter context is generated server-side at `POST /encounter/start` and persisted in the character save file; it contains a player-facing `introduction` (shown in the overlay) and a behind-the-scenes `context` string available to AI agents. The server also generates all encounter content at this point: `enemyCount`, `secrets`, `riddle`, and `quests`.
- **US-029** As a player, I can select from a list of premade encounters so that I can replay encounters I enjoy. The Encounter Setup Scene shows a two-column layout: **Character** (left, ~920 px wide) renders one 270 px card per file in `server/data/characters/` (currently three cards fit side by side); **Encounter** (right, ~780 px wide) holds a 2-column grid of 360 px encounter cards. Each encounter card shows the map name, title, encounter-type chips (colour-coded: Combat red, Exploration green, Social blue), and a flavour description. Premade encounters are JSON files in `server/data/encounters/` served via `GET /encounters`; each entry carries `mapId`, `encounterTypes`, and optional fields: `npcIds` (NPCs to spawn), `allyIds` (NPCs spawned with ally disposition near the player), `customIntroduction` (overrides the auto-generated introduction text), `customContext` (overrides the auto-generated AIGM context), `tileProperties` (per-GID `passable` overrides), and `startingZones` (a Tiled-style tile layer with implicit spawn-zone GIDs: 1 = player, 2 = ally, 3 = neutral NPC, 4 = enemy). Adding an encounter is a JSON drop with no code change; the shipped set is content, not a requirement. The BEGIN ENCOUNTER button activates once a character and an encounter are both selected.
- **US-031** As a player, I can open a conversation with an AI Game Master (AIGM) at any time during an encounter so that I can interact with the world through free-form text. The AIGM is accessible via the **GAME MASTER** tab in the HUD (inline, not a separate overlay). The player types into the input box at the bottom of the tab; a mode button to the left of the input switches between **GM** (default — message goes directly to the GM) and **Say to [Name]** (in-character — the message is prefixed `[PlayerName says to Target]:` before being sent; only available when a living NPC is selected; the button label updates to the NPC's revealed name once known). The Story GM is powered by Claude Sonnet; the Dev GM uses Claude Haiku. The GM reply is **streamed** to the chat panel over the WebSocket via `aigm_chunk` messages, so text appears live as Claude generates it. Chunks written before a roll-requesting tool resolves are speculative; the server rolls them back via `aigm_speculative_discard` and the real text follows after the roll. Chunks accompanying non-speculative tools are confirmed with `aigm_checkpoint`. A final `aigm_done` carries the canonical reply and any roll-result strings (rendered as 🎲 entries). The AIGM receives complete, real-time encounter state on every message: player stats (HP, XP, gold, AC, inventory, hidden status, action/bonus used, moves left, class-feature resources, spell slots, concentration target), all living combatants (enemies and allies: HP, AC, position, active-turn flag, vexed/hidden status, disposition, attacks, revealed name if any), all living neutral NPCs (defId, entity ref, position, revealed name if any), dead NPCs as a separate CORPSES section (name, position — searchable but cannot act), quest progress, items on the ground, secrets remaining, the selected target, and the full event log. The AIGM can control all aspects of the encounter via tool use: adjust any entity's HP (`adjust_npc_hp`: entity format `"player"`, `"enemy_A"`, or `"npc_[id]"`; positive delta heals, negative damages; kills the entity when HP reaches 0), award XP/gold, add narrative log entries, teleport entities (`move_entity`), give the player items, remove items from inventory, start or end combat, force-complete quests, toggle player stealth, spawn a new enemy onto the map, despawn an NPC, change any NPC's disposition (`set_disposition`), apply or remove conditions, throw any inventory or map item at an enemy (`throw_item`: proper thrown weapons use weapon stats; other items are improvised 1d4 bludgeoning), reveal an NPC's name (`reveal_npc_name`: sets `NpcState.revealedName` and updates the token name label; must be called before narrating the name so the UI reflects the reveal), mark an ally as combat-passive (`set_npc_passive`: when `passive: true` the ally skips their combat turn entirely; reversed with `passive: false`; used when the player tells an ally to stand down or stay back), and search the full conversation archive for older content (`recall_memory`: case-insensitive substring search, returns up to 8 newest-first matching snippets — used when the sliding-window history has been summarised and the model needs to look up an earlier exchange), durably shift the player's reputation with a faction (`adjust_faction_standing`: writes `GameState.factionStandings`, clamped to [−100, +100], surfaced to the GM in CURRENT STATE under FACTION STANDINGS), and record a significant world event into long-term world memory (`create_rumor`: appends to `GameState.rumors` idempotently by id, surfaced to the GM under the RUMORS block — see US-069). The entity ref format for all tools is: `"player"`, `"enemy_A"` (enemy by `combatLabel`), `"ally_A"` (ally by `combatLabel`, or `"npc_[id]"` when no label assigned), `"npc_[id]"` (neutral or ally by id). The selected target is surfaced to the AIGM with its entity ref in the focus line. When `trigger_combat` is called in a social encounter with no enemies, all present NPCs are converted to enemy combatants in-place. When moving multiple entities to the same tile, each is placed on the nearest free passable tile. The AIGM roleplays all NPCs in their own voice, constrained by their persona and knowledge. Pronouns ("them", "it") are resolved to the player's currently selected target. Conversation history is stored server-side in the session and preserved for the duration of the encounter; the client retains a local copy for display. To keep cost and latency bounded the server maintains two histories per session: a **sliding window** (`aigmHistory`) of the most recent ~20 messages that is what's actually sent to Claude on each turn, and a **full archive** (`aigmArchive`) of every exchange that's only consulted by the `recall_memory` tool. When the sliding window exceeds 40 messages the oldest entries are collapsed into a single `[SUMMARY OF EARLIER TURNS]` assistant message generated by Claude Haiku, and the most recent 20 messages are kept verbatim. The full archive is in-memory only and resets if the server restarts. The system prompt is split into a static instructions section (eligible for Anthropic prompt caching) and a dynamic state section injected into each user message. Historical `[CURRENT STATE]` blocks are stripped from the conversation history before each API call so the model always reasons from the freshly injected state. Both the system prompt and the tool list carry `cache_control: ephemeral` markers, and inside the tool-use loop the most-recent `tool_result` block also gets a cache_control marker so long tool chains don't re-pay the prefix. The loop is capped at 8 iterations: on the final iteration every tool result is overridden with a `TOOL BUDGET EXHAUSTED` signal and a tool-less follow-up forces the model to finalize its narrative. Each Claude API call is wrapped in a single retry-with-backoff on transient errors (408, 425, 429, 5xx, 529). Concurrent AIGM requests on the same session are rejected with HTTP 429 via a per-session mutex. The GM operates under these constraints: **TOOL-FIRST RULE** (every game effect must be enacted via the corresponding tool before it is narrated; `request_attack_roll` does not apply damage — the GM must follow up with `adjust_npc_hp`; `set_disposition` does not start combat on its own — the GM must call `trigger_combat` after disposition changes in the exploring phase; NPC name reveals require `reveal_npc_name` before the name appears in narration); **NPC NAMES** (when any NPC gives their name, `reveal_npc_name` must be called first with the entity ref from CURRENT STATE and the chosen name); **WORLD GROUNDING** (only references creatures, items, and events present in state or established in conversation); **TURN ORDER** (during `player_turn`, does not narrate or simulate enemy turns); **PLAYER AGENCY** (enacts any mechanically possible action if the player confirms after one warning); **STATE IS AUTHORITATIVE** (trusts current state over conversation history without comment); **SEARCHING CORPSES** (`request_ability_check` before narrating what is found). Gold transactions require `award_gold` before narration. Skill-based player actions require `request_ability_check`. The CURRENT STATE block also surfaces queued **SCRIPTED EVENTS** (lines pushed by `send_aigm_message` trigger actions — incorporate into the next reply, then the server clears them) and, when set, **FACTION STANDINGS** and **RUMORS** blocks for long-term world memory. The GM operates **out of the action critical path**: `POST /game/session/:id/action` resolves every player action through the deterministic engine without ever calling Claude, and the client's `DevMode.disableAIGM` flag (URL param `?disableAIGM=true` or localStorage) short-circuits the GM Chat callback so the encounter is fully playable on the deterministic layer alone (see US-068).
- **US-032** As a player, I want to find and equip gear with stats so that my equipment choices meaningfully affect gameplay. Each character's `defaultEquipment` field in their JSON declares the starting loadout (which item ids start in inventory and which slots they fill); the engine applies it at session start. Equipment is split into three slot types — armor, weapon, offhand (shield) — stored in `EquipmentSlots`. Armor AC is computed per SRD category (light: base+DEX; medium: base+min(DEX,2); heavy: fixed base) plus +1 for Fighting Style: Defense and +2 for a shield. Finesse weapons use the higher of STR/DEX for attack and damage. Character stats that depend on data objects are never stored in character JSON files — they are always computed at startup and session creation by three functions that run in order: (1) `applySpecies` reads `speciesId` and `speciesLineage` to derive `speed` (base species speed + any lineage bonus); (2) `applyFeats` reads `featIds` to derive `savageAttacker` and `fightingStyleDefense`; (3) `applyEquipment` reads `defaultEquipment` to derive `ac` and `mainAttack`. The Character Sheet Overlay's Inventory tab is opened from the CHARACTER button in the Player Panel — it shows the three equipped slots with computed stats and UNEQUIP buttons, a scrollable carried-items list with EQUIP buttons (identical items grouped with a ×N count), and a live AC + attack summary bar. Equipping a two-handed weapon auto-unequips any held shield. Equipping one item from a stack removes only that instance. Equipment changes broadcast to the client via `PlayerState.ac` so the displayed AC updates immediately. Equipped items are persisted in the save file. The AIGM is aware of equipped armor, weapon, and shield and can grant or remove items via `add_item` and `remove_item`.
- **US-033** As a player, I want to see my save information inside the character selection card so that I know where each character left off. Each character card on the **Encounter Setup Scene** and the **Adventure Setup Scene** shows HP / XP / GP from the saved state and the currently equipped weapon and armor. A DELETE SAVE button inside the card clears the save from localStorage and the server without leaving the setup screen — on the Adventure Setup Scene it also clears the adventure save for that character (so the player can replay the adventure from chapter 1 with default gear). On load, each card always syncs from the server: when the server returns a save it overwrites the local mirror and refreshes HP/XP/GP, equipped gear, encounter log, and story log in place; when the server returns `null` (no save file on disk) the local mirror is purged and the card resets to a "no save" state — DELETE SAVE and STORY LOG are dimmed and disabled. The `GET /save/:characterId` route uses `readSaveIfExists` so missing saves return `null` rather than a fabricated default, preventing stale localStorage from making freshly-deleted saves look active. The last-played character's card is pre-selected on boot. Distinct from clearing a save, a **DELETE CHARACTER** button beneath the carousel (hidden while the Create card is focused) permanently removes the whole character: it opens a confirmation modal and, on confirm, deletes the character definition (`DELETE /characters/:id`) and its save before refreshing the roster and rebuilding the scene.
- **US-038** As a player, I can open my character sheet to see my stats, manage gear, and review prepared spells without leaving the encounter. A CHARACTER button in the Player Panel opens the Character Sheet Overlay with five tabs: **Stats** (HP / AC / Speed / Initiative / Proficiency, ability scores with modifiers, saving throws with proficiency dots, XP / gold / passive perception, concentration chip when active), **Features** (walks the class progression L1..currentLevel + subclass progression and renders each granted feature with its SRD description and the level it was gained at — see US-091), **Story** (per-encounter story log entries — see US-059), **Equipment** (three equipped slots with UNEQUIP buttons + scrollable carried-items list grouped by category — equippable / consumables / ammunition / gear — with EQUIP, USE, AMMO, and GEAR controls; stats bar with AC, GP, and main attack summary), and **Spells** (caster only — Save DC / Spell Attack / slot pool header; Cantrips / Prepared / Spellbook · Unprepared sections; per-row CAST and RITUAL CAST buttons). State updates trigger a live rebuild so equip / cast / damage feedback shows up without reopening. AC displays the boosted value plus "(Mage Armor)" while the spell is active.
- **US-039** As a player, I can choose between multiple AIGM personas so that I can tailor the game to my mood. Two personas are available, toggled via STORY / DEV chips in the GM overlay header. **Story GM** (default): enforces SRD 5.2.1 rules and in-world logic. It operates on a tool-first rule: every game effect it narrates must be enacted via the corresponding tool before it is described — weapon throws call `throw_item`, damage to any entity calls `adjust_npc_hp`, movement calls `move_entity`, and so on. If an effect cannot be enacted with the available tools, the GM does not narrate it as happening. It explicitly rejects requests that would require conjuring items or creatures from nothing — `add_item` and `spawn_enemy` are only used when the fiction justifies it (finding loot, triggering an ambush), never simply because the player asks. It redirects rejected requests toward a realistic in-world alternative. It never uses meta phrases about dice mechanics — roll results appear via the 🎲 overlay entry and the GM narrates only the in-world outcome. **Dev GM**: intended for development and testing — fulfils all requests without restriction, using any tool needed, and replies with brief mechanical feedback (which tools were called and what changed) rather than narrative prose. The selected persona is sent to the server on every message and persists across open/close cycles for the duration of the encounter.
- **US-040** As a player, I can get a user interface that automatically fits the available space, so I can play the game on different browser window sizes. Phaser's Scale Manager runs in `FIT` mode with `CENTER_BOTH`: the canvas renders at a fixed logical resolution (1900 × 880) and is CSS-scaled to fill the browser window while preserving the aspect ratio. All UI panels and overlays (Player Panel, Target Panel, HUD, Turn Order Bar, and all overlays) are HTML DOM elements positioned and scaled over the canvas by `UIScale` (`client/src/ui/UIScale.ts`), which uses a single `transform: scale()` per panel derived from the canvas bounding rect. A `ResizeObserver` on the canvas re-positions all panels whenever the display size changes. The game world (tiles, tokens, sprites) remains on the Phaser canvas.
- **US-056** As a developer, I want feats, backgrounds, and species from SRD 5.2.1 chapter 04–05 stored as structured data objects so that character definitions reference them by ID rather than duplicating their mechanical effects. All 17 feats (`server/data/feats/`), 4 backgrounds (`server/data/backgrounds/`), and 9 species (`server/data/species/`) are implemented as JSON files. Each feat has a `category`, `prerequisites`, `description`, and a structured `effects` object keyed by named mechanical properties. Each background records its ability scores, granted feat, skill and tool proficiencies, and two equipment options. Each species records creature type, size, speed, and an array of traits — each trait carries a `description` and a structured `effects` object. Characters reference these by `speciesId`/`speciesLineage`, `backgroundId`, and `featIds[]`. Two mechanical effects are currently wired into the engine: `applyFeats` derives `savageAttacker` and `fightingStyleDefense` from `featIds`; `applySpecies` derives `speed` from the species base speed plus any lineage speed bonus. The Fighting Style level-up picker introduced by US-091 selects from feats whose `category` is `'fighting-style'` (Defense, Archery, Two-Weapon Fighting, Great Weapon Fighting), but currently only Defense's `armorAcBonus` is consumed by the engine — the other three feats land in the character's `featIds` for future wiring. All three collections are served via REST (`GET /feats`, `GET /backgrounds`, `GET /species`) and cached in the client registry.
- **US-059** As a player, I can read a Story Log which gives me an AI-generated prose recap of every encounter I have completed, so that I can build a narrative record of my character's adventures. A STORY LOG button on the character card in the Encounter Setup Scene opens the Story Log Overlay. The overlay lists all completed encounters in chronological order; each entry shows the date, encounter type, and a short prose narrative generated by Claude Sonnet. The narrative is written in third-person, two to three sentences long (one short paragraph), weaves the encounter description naturally into the story, translates mechanical events into fiction (hits become blows landed, kills become deaths), includes meaningful dialogue in italics, and omits all mechanical numbers. Narratives are generated on demand via `POST /save/:characterId/storylog`, which processes only encounters not yet in the log and appends new entries — old entries are never rewritten. A GENERATE N ENTRIES button in the overlay triggers generation and is disabled when the log is already up to date. A REWRITE ALL debug button regenerates all entries from scratch. Generated narratives are stored in the character save under `storylog` and persist across sessions. The generation logic lives in `server/src/storylog.ts`.
- **US-064** As a player, I want the world built from tilemaps so that levels are visually rich and easy to author. Hand-crafted maps are stored in **Tiled-compatible JSON** (`server/data/maps/*.json`) referencing the in-tree Kenney Roguelike Pack atlas (`server/data/tilesets/roguelike.png` + `roguelike.tsj`, 16×16 tiles, 1px spacing, 57×31 grid) plus an AI-facing tile legend (`roguelike_legend.json`) that documents each GID's name, default passability, and intended layer. Maps may carry **two stacked tile layers**: a `terrain` (ground) layer plus an optional `objects` layer drawn on top; a cell is passable iff both layers' GIDs are passable. The server resolves external `source: "../tilesets/xxx.tsj"` references and surfaces tileset metadata + image URLs to the client under `GameMap.tilesets[]`. `BootScene` queues every unique tileset image as a Phaser spritesheet on a second loader pass before transitioning to `GameScene`, which then renders each tile in `drawMapTiles` via `this.add.image(..., tilesetTextureKey(ts.imageUrl), gid − firstgid)`. Map sprites render at `MAP_TILE_ALPHA = 0.7` so the dark scene background bleeds through to darken the overall map. Maps without an image tileset (the procedurally generated ones) keep the older coloured-fill rendering.
