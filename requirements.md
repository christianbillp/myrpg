# MyRPG — Requirements

## Overview

A browser-based 2D single-player RPG built with modern web technologies.

## Tech Stack

| Layer            | Technology        | Version                  | Status  |
| ---------------- | ----------------- | ------------------------ | ------- |
| Game engine      | Phaser.js         | 3.88.x                   | In use  |
| Language         | TypeScript        | 5.7.x                    | In use  |
| Build tool       | Vite              | 6.3.x                    | In use  |
| Backend          | Node.js + Fastify | Node 22 LTS, Fastify 5.x | In use  |
| Database         | PostgreSQL        | 17                       | Planned |
| Cache / sessions | Redis             | 7                        | Planned |

## Functional Requirements and Roadmap

### Core Gameplay

### Done

- **US-002** As a player, I want to choose between Aldric Vane (Human Fighter) and Miriel Duskwhisper (Wood Elf Rogue) before the encounter begins so that I can experience different combat styles. Each character has distinct mechanics: Aldric uses Greatsword with Savage Attacker and Graze mastery plus Second Wind; Miriel uses Shortsword with Sneak Attack (from hiding), Vex mastery, and a Hide action.
- **US-001** As a player, I want to move my character around the world using keyboard input (WASD and arrow keys) so that I can explore the game world. The world is represented as a tile-based map where each tile equals 5 feet, making movement structured and tile-aligned.
- **US-005** As a player, I want to engage in turn-based combat with enemies so that exploration feels challenging and rewarding. Combat is fully integrated into the map: movement (30 ft = 6 tiles per turn) and actions happen on the same map, following SRD 5.2.1 rules (initiative, attack rolls, Greatsword with Savage Attacker and Graze mastery, Second Wind, Nimble Escape, death saving throws). The combat log is a two-column display: the left column shows immersive narrative ("Hit — 8 slashing"), the right column shows dice detail ("d20(14)+5=19 vs AC 14 · 1d6[5]+3"). Each row is colour-coded by outcome. It is possible to scroll through the entire combat log.
- **US-017** As a developer, I want every creature's stat block to be sourced directly from the D&D 5e SRD 5.2.1 so that combat mechanics are correct and new creatures can be added without guesswork. Monsters (`MonsterDef`) hold SRD stat blocks: creature type, HP dice formula, full ability scores, an explicit `initiativeBonus` field, and attack entries with attack type (melee / ranged / both), reach, and range. NPCs (`NPCDef`) are a separate type that references a `monsterClass` for their stat block while adding identity fields (name, colour, persona). The combat system selects the correct attack type based on range. Monster roster: Goblin Minion, Bandit, Commoner, Skeleton (CR 1/4, AC 14, bludgeoning vulnerability, Shortsword + Shortbow).
- **US-019** As a player, I want to move around on a randomly generated map so that I can experience a variety of challenges. Each encounter generates a new square map with a random side length between 10 and 30 tiles, where 20–40% of tiles are impassable walls. All passable tiles are guaranteed to be reachable from each other (flood-fill validated). Player and enemies spawn on passable tiles separated by at least 5 tiles. Wall tiles are visually distinct. Movement respects walls including the SRD diagonal corner rule (diagonal movement is blocked when both orthogonal neighbours are walls). The enemy AI uses BFS pathfinding to navigate around walls.
- **US-006** As a player, I want to pick up, manage, and use items in an inventory so that I can equip myself for challenges. Up to 3 Health Potions (2d4+2 healing per SRD) spawn as green diamond tokens on the map each encounter; walking onto a token picks it up automatically. Gold pieces (GP) are awarded on kill at 10 × CR (rounded down). Both XP and GP rewards are recorded in the combat log. Current GP is shown in the Inventory Overlay stats bar.
- **US-021** As a player, I want to select a Social Interaction encounter so that I can engage with an NPC rather than fight. All creature communication — with NPCs and enemies alike — goes through the Dungeon Master overlay (Claude Sonnet). The player opens the DM overlay and addresses any creature by name or pronoun; the DM responds in that creature's voice, constrained by its persona and knowledge. Enemies may taunt, plead, or bargain. The DM calls `complete_quest('make_contact')` automatically when the player first successfully engages an NPC in a social encounter.
- **US-022** As a player, I want to select the Exploration encounter type so that I can discover hidden secrets on the map. When active, 4 secrets are placed on the map (invisible to the player). A SEARCH button appears in the Player Panel during exploration while secrets remain. The HUD phase text appends "· search available" as a hint. Using Search performs a Wisdom (Perception) check (d20 + perception bonus vs secret DC 10/12/15). While adjacent to a secret tile, success reveals the reward (gold, a Health Potion, or a lore inscription) and logs it; failure produces a flavour observation. Each secret can only be found once. The SEARCH button is hidden when no secrets remain.
- **US-023** As a player, I want to play on hand-crafted saved maps so that I can experience specific layouts. Six maps are available: Arena, Dungeon, Ruins, Catacombs, Forest Trail, and The Rusty Flagon (tavern), stored as JSON in `server/data/maps/` and served via `GET /maps`. Each map is pre-processed server-side into a `passable` boolean grid. Map selection is implicit in the premade encounter choice (each `PremadeEncounterDef` carries a `mapId`); the standalone map picker overlay is no longer used.
- **US-024** As a player, I want to fight multiple enemies in a single combat encounter so that battles feel more strategic. Simple Combat spawns 2–4 enemies chosen randomly from available enemy types (Goblin Minion and Bandit). All map enemies join combat the moment one triggers it. Enemies are labeled A, B, C… on the map and in the Turn Order Bar at the top of the Game Map. Enemies act sequentially during the enemy phase; the active chip in the Turn Order Bar is highlighted green and dead chips are dimmed. Combat ends only when all enemies are defeated.
- **US-025** As a player, I want the action economy to follow D&D 5e SRD rules so that combat feels authentic. Each player turn grants one Action and one Bonus Action. Attack consumes the Action; Second Wind (Fighter) and Hide via Cunning Action (Rogue) consume the Bonus Action. Drinking a Health Potion costs the Bonus Action in combat and is free during exploration. The player remains in their turn after spending either resource and must press End Turn to hand initiative to the enemies. The Phase Text appends "· action used" or "· bonus used" when the respective resource is spent; buttons that require a spent resource are dimmed and disabled.
- **US-007** As a player, I want to receive and track quests so that I have clear goals and a sense of progression. Quests are assigned automatically at encounter start based on the active encounter types. Simple Combat assigns: First Blood (slay 1 enemy, +10 XP +5 GP), Slay All (slay every enemy spawned, +25 XP +15 GP), and Treasure Hunt (collect 2 Health Potions, +10 XP +5 GP). Exploration assigns: Keen Eye (find 2 secrets, +15 XP +10 GP). Social Interaction assigns: Make Contact (open a conversation with any NPC, +10 XP +5 GP, awarded on the first AI reply or riddle answer). The Player Panel shows a QUESTS section listing each quest as "· Title N/M" while in progress and "✓ Title" when complete. Quest rewards are added to XP and GP and logged in the Combat Log on completion.
- **US-008** As a player, I want my game state saved and loaded automatically so that I can continue where I left off. Save state is split into two files on the server: a **character save** at `server/data/saves/{characterId}.json` (HP, XP, gold, inventory, equipment, Second Wind uses) and a **world save** at `server/data/saves/world.json` (full session/encounter state, referencing the character by `defId`). On boot, `BootScene` calls `GET /world`; if a world save exists the session is restored and `GameScene` opens directly, otherwise `EncounterSetupScene` is shown. The character save is written on every action and after every AIDM exchange. The world save is deleted when the player presses NEW ENCOUNTER. Each character card in the setup screen shows HP / XP / GP and equipped gear from the character save; a DELETE SAVE button clears the character save without leaving the setup screen.
- **US-026** As a developer, I want all game content served from the Fastify REST API so that content can be updated centrally without redeploying the client. Static JSON files live under `server/data/` in per-type subdirectories. The server exposes `GET /characters`, `GET /monsters`, `GET /npcs`, `GET /items`, `GET /maps`, and `GET /premade-encounters`; each endpoint reads all `.json` files in its directory and returns them as an array. The client preloads all six collections in `BootScene` and stores them in the Phaser registry.
- **US-027** As a developer, I want NPCs to be a distinct data type from monsters so that non-combat characters can carry dialogue personas without polluting SRD stat blocks. NPCs live in `server/data/npcs/` as `NPCDef` objects with `id`, `name`, `monsterClass` (reference to a `MonsterDef` id for stat resolution), `color`, and an optional `persona` string. At spawn time the client merges the NPC's identity over the referenced monster's stat block. The `POST /npc/chat` endpoint looks up the NPC by id, resolves its persona, and forwards the conversation to the Claude API. Current roster: Wandering Sage (`wanderer`), Local Villager (`villager`), Bram Holdfast the tavern keeper (`tavern_keeper`).
- **US-004** As a player, I want to select an AI Dialogue encounter so that I can have a freeform conversation with an AI-powered NPC. Superseded by US-021; the `ai_dialogue` encounter type has been removed and merged into `social_interaction`. All NPC encounters now use AI dialogue by default with riddle fallback.
- **US-037** As a player, I want my full skill list stored and used in play so that ability checks reflect my character's actual proficiencies and create meaningful risk. Each character's `skills` map (stored in the character JSON and persisted in `SaveData`) holds all 18 SRD skills keyed by camelCase name (e.g. `sleightOfHand`). Each value is the pre-computed total modifier: ability modifier plus proficiency bonus once for proficiency, twice for expertise, or not at all if untrained. The dedicated `perceptionBonus` and `stealthBonus` fields on `PlayerDef` are replaced by the general map. Skills are surfaced to the AIDM on every message. When the player asks to do something that does not automatically succeed, the AIDM calls a `request_ability_check` tool specifying the skill, a DC (Very Easy 5 / Easy 10 / Medium 15 / Hard 20 / Very Hard 25 per SRD), and a reason. The server rolls d20 + the skill modifier and returns the result; the AIDM overlay injects it as a distinct 🎲 roll entry (green on success, red on failure) between the player's message and the DM's narrative response. The AIDM then narrates success or failure and uses its other tools to apply consequences — awarding or deducting HP, XP, or gold, moving entities, or altering the encounter — as the fiction demands.
- **US-030** ~~Superseded.~~ Passive NPCs removed. All NPCs are the same type; each encounter specifies zero or more NPCs via the `npcIds` array. Multiple NPCs of the same type (e.g. several villagers) can be included by repeating the same id.
- **US-036** As a player, I want saving throws tracked for all creatures so that the AIDM can call for saves when the story demands it. Every creature carries a `savingThrows` map of all six modifiers: character JSONs derive these at load time from `savingThrowProficiencies` (ability modifier + proficiency bonus if proficient); monster JSONs store the values directly from the SRD stat block SAVE column. All six player saving throws are surfaced to the AIDM on every message. The AIDM actively calls for saving throws — against traps, spells, environmental hazards, or any narrative moment where a save is appropriate — using the same `request_ability_check` flow as skill checks, rolling d20 + the relevant save modifier, then narrating success or failure and applying consequences via its other tools.
- **US-041** As a player, I want enemies to make Opportunity Attacks when I move out of their reach, and I want to make one when an enemy moves away from me, so that positioning carries meaningful risk. When a creature uses its movement to leave the reach of an enemy that can see it, that enemy may spend its Reaction to make one melee attack. Each creature has one Reaction per round, reset at the start of its turn. The player's Opportunity Attack fires automatically when an enemy moves away. Enemy Opportunity Attacks fire as the player moves through a threatened tile. The combat log notes each Opportunity Attack.
- **US-042** As a player, I want Dash, Dodge, and Disengage actions so that I have tactical options beyond attacking. **Dash**: spend Action to gain extra movement equal to Speed for the turn. **Dodge**: until start of next turn, attacks against the player have Disadvantage and the player has Advantage on DEX saves; shown in Phase Text. **Disengage**: movement does not provoke Opportunity Attacks for the rest of the turn. These appear as Action Buttons in the Player Panel when the Action has not yet been spent.
- **US-043** As a developer, I want a formal Advantage/Disadvantage system so that all D20 Test modifiers are consistent and composable. Any D20 Test can be tagged with Advantage or Disadvantage sources. When resolving the roll: Advantage → roll 2d20, take higher; Disadvantage → take lower; both present → cancel, roll normally. Sources include: player hidden (Advantage on attack), Vex condition (Disadvantage on target's next attack), Dodge (Disadvantage on attacks against dodger), Surprise (Disadvantage on surprised creature's Initiative). The combat log shows both dice values when the mechanic applies.
- **US-051** As a developer, I want a formal conditions framework so that status effects are consistent, visible, and extensible. Conditions (Prone, Poisoned, Frightened, Grappled, Blinded, Stunned, Invisible, etc.) are tracked as a set per creature. Each condition's mechanical effects are applied at resolution time by the combat and movement systems. The Target Panel shows active conditions on the selected creature. The AIDM can apply and remove conditions via `apply_condition` and `remove_condition` tools.
- **US-047** As a developer, I want Passive Perception implemented so that hiding is not an automatic success. Passive Perception = 10 + Perception modifier. A player's Hide check must exceed the highest Passive Perception among all living enemies to succeed; failure is logged and the player remains visible.
- **US-049** As a developer, I want Massive Damage / Instant Death implemented. When a hit reduces the player to 0 HP and leftover damage ≥ player's HP maximum, the player dies instantly without death saving throws. The combat log notes the instant death.
- **US-053** As a developer, I want monster Resistance and Vulnerability modeled so that damage is applied correctly per SRD stat blocks. Each `MonsterDef` carries optional `resistances` and `vulnerabilities` arrays of damage types. Player attacks check the target's lists and halve (resistant) or double (vulnerable) damage before applying to HP, with a log entry noting the modifier. Each `PlayerAttack` now carries a `damageType` field sourced from the equipped weapon (or "bludgeoning" for Unarmed Strike).
- **US-046** As a player, I want to take a Short Rest after an encounter so that I can recover HP between fights by spending Hit Dice. A SHORT REST button appears in the Player Panel during the exploring phase when the player is below max HP and has Hit Dice remaining. Each click spends one Hit Die (d10+CON for Fighter, d8+CON for Rogue) and heals that amount, logged in the Combat Log. Hit Dice equal the character level; they reset at the start of every new encounter (Long Rest equivalent).

### Next

- **US-044** As a player, I want some map tiles to be Difficult Terrain so that terrain shapes movement. Difficult Terrain costs 2 ft per foot moved. Map files and the generator can designate difficult terrain tiles. Movement highlights account for terrain cost.
- **US-045** As a player, I want cover to protect targets near walls so that positioning near obstacles is tactically meaningful. Half Cover (+2 AC and DEX saves), Three-Quarters Cover (+5 AC and DEX saves), Total Cover (untargetable). Cover is determined geometrically at attack resolution.
- **US-048** As a player, I want Surprise to apply when combat starts unexpectedly. Enemies that are unaware when combat begins (e.g. player is hidden) roll Initiative with Disadvantage. If an ambush catches the player unaware, the player rolls Initiative with Disadvantage. The combat log notes surprised combatants.
- **US-050** As a player, I want a Shove option so that I can knock enemies Prone. Shove consumes the Action. The enemy makes a STR or DEX saving throw (DC = 8 + player STR mod + proficiency bonus). On failure the enemy has the Prone condition: Disadvantage on its attack rolls; melee attacks against it have Advantage, ranged attacks have Disadvantage. Rising from Prone costs half Speed.
- **US-052** As a player, I want to knock enemies unconscious instead of killing them so that I can resolve encounters non-lethally. When a melee attack would reduce an enemy to 0 HP, the player can choose "Knock Out". The enemy has the Unconscious condition (Incapacitated, Prone, Speed 0, Advantage on attacks against it, auto-crit if adjacent) and is removed from the initiative order.

- **US-034** As a player, I want to throw javelins so that I have a ranged attack option in combat. The `javelin` item JSON (1d6 piercing, STR-based, Mastery: Slow) exists and Aldric carries 8 by default. What remains is a ranged ATTACK action on the player's turn that resolves using thrown range rather than melee reach. The enemy AI already selects attack type by range; the player side does not yet have an equivalent.
- **US-035** As a developer, I want all SRD weapon mastery properties implemented so that weapons behave correctly. Sap (impose disadvantage on the target's next attack roll) and Slow (reduce target speed by 10 ft until start of your next turn) are used by Aldric's flail and javelin but are absent from the `WeaponMastery` type. This requires defining their effect logic alongside the existing Graze and Vex implementations and wiring them into the combat system.

### User Interface

### Done

- **US-040** As a player, I can get a user interface that automatically fits the available space, so I can play the game on different browser window sizes. Phaser's Scale Manager runs in `FIT` mode with `CENTER_BOTH`: the canvas renders at a fixed logical resolution (1700 × 950) and is CSS-scaled to fill the browser window while preserving the aspect ratio.

- **US-003** As a player, I want to see my character's stats in a toggleable sidebar so that I can inspect my current HP, AC, speed, ability scores, and XP without interrupting play. The Player Panel displays name, species/class/level, a colour-coded HP bar, combat stats (AC, Speed, Proficiency bonus, Initiative), all six ability scores with modifiers, current XP, quests, context-sensitive combat action buttons (Attack, Dash, Dodge, Disengage, Second Wind, Hide, End Turn, Roll Death Save — visible only when the relevant conditions are met during combat), and the persistent buttons INVENTORY and SEARCH. The panel is hidden by default and toggled open/closed by clicking the player token on the Game Map.
- **US-016** As a player, I want to click a creature in the game map to select it so that I can inspect its stats. The selected creature is highlighted with a coloured outline. A Target Panel appears on the right side of the Game Map showing the creature's name, HP bar, AC, speed, and ability scores — mirroring the layout of the Player Panel. Clicking elsewhere or defeating the creature deselects it and hides the panel. When combat starts, the nearest active enemy is automatically selected.
- **US-018** As a player, I want to zoom in and out on the game map using the mouse wheel so that I can get a better view of the battlefield. I also want to pan around the map by left-click dragging.
- **US-020** As a player, I want to be presented with an encounter setup menu before the game starts so that I can choose my character and encounter. Superseded by US-029; the setup scene now shows a two-column layout (Character | Premade Encounter) rather than the original three-column free-form selector.
- **US-029** As a player, I can select from a list of premade encounters so that I can replay encounters I enjoy. The Encounter Setup Scene shows a two-column layout: Character (left) and Premade Encounter (right — a 2×4 grid of encounter cards). Each card shows the map name, title, encounter-type chips (colour-coded: Combat red, Exploration green, Social blue), and a flavour description. Premade encounters are JSON files in `server/data/premade-encounters/` served via `GET /premade-encounters`; each entry carries `mapId`, `encounterTypes`, and an optional `npcIds` array (specifying which NPCs to spawn). Eight encounters ship: Arena Brawl, Dungeon Crawl, Ruins Mystery, Catacomb Hunt, The Sage's Counsel, Full Assault, Forest Ambush, A Night at the Flagon. The BEGIN ENCOUNTER button activates once a character and an encounter are both selected.
- **US-028** As a player, I get an introduction to the encounter when I start it so that I feel more immersed in it. When the game map loads, an Introduction Overlay appears before the player can act. It shows the active encounter type chips, the player's name and class, and a narrative introduction paragraph. The overlay must be dismissed before play begins. The encounter context is generated server-side at `POST /encounter/start` and persisted in the character save file; it contains a player-facing `introduction` (shown in the overlay) and a behind-the-scenes `context` string available to AI agents. The server also generates all encounter content at this point: `enemyCount`, `secrets`, `riddle`, and `quests`.
- **US-038** As a player, I can open my inventory to see and manage all carried items so that I can equip gear and use consumables without leaving the encounter. An INVENTORY button in the Player Panel opens the Inventory Overlay showing three equipped slots (Armor, Weapon, Offhand) with UNEQUIP buttons, and a scrollable carried-items list. Identical items are grouped with a ×N count. Equippable items show an EQUIP button; consumables show a USE button (dimmed when a bonus action cannot be spent). When content exceeds the visible area the list scrolls with the mouse wheel and a scrollbar thumb tracks position. A stats bar at the bottom shows AC, current GP, and the main attack summary.
- **US-039** As a player, I can choose between multiple AIDM personas so that I can tailor the game to my mood. Two personas are available, toggled via STORY / DEV chips in the DM overlay header. **Story DM** (default): focuses on narrative immersion and enforces the SRD 5.2.1 rules — it fulfils creative roleplay requests freely but rejects out-of-rules actions (e.g. "teleport me", "kill all enemies", "give me 1000 gold"), redirecting the player toward a narrative path instead. It never uses meta phrases about dice mechanics ("let's see", "rolling now", etc.) — roll results are shown via the 🎲 overlay entry and the DM narrates only the in-world outcome. **Dev DM**: intended for development and testing — fulfils all requests without restriction, including moving the player, removing enemies, and granting arbitrary XP/HP/gold. The selected persona is sent to the server on every message and persists across open/close cycles for the duration of the encounter.

### Player Progression

### Done

- **US-032** As a player, I want to find and equip gear with stats so that my equipment choices meaningfully affect gameplay. Each character has a default loadout sourced from their character sheet (Aldric: Chain Mail + Greatsword equipped, Flail + 8 Javelins carried; Miriel: Leather Armor + Shortsword equipped, 4 Daggers + Shortbow carried). Equipment is split into three slot types — armor, weapon, offhand (shield) — stored in `EquipmentSlots`. Armor AC is computed per SRD category (light: base+DEX; medium: base+min(DEX,2); heavy: fixed base) plus +1 for Fighting Style: Defense and +2 for a shield. Finesse weapons use the higher of STR/DEX for attack and damage. An INVENTORY button in the HUD opens the Inventory Overlay, which shows the three equipped slots with computed stats and UNEQUIP buttons, a scrollable carried-items list with EQUIP buttons (identical items grouped with a ×N count), and a live AC + attack summary bar. Equipping a two-handed weapon auto-unequips any held shield. Equipping one item from a stack removes only that instance. The Player Panel AC stat updates live whenever gear changes. Equipped items are persisted in the save file. The AIDM is aware of equipped armor, weapon, and shield and can grant or remove items via `add_item` and `remove_item`.
- **US-033** As a player, I want to see my save information inside the character selection card so that I know where each character left off. Each character card in the Encounter Setup Scene shows HP / XP / GP from the saved state and the currently equipped weapon and armor. A DELETE SAVE button inside the card clears the save from localStorage and the server without leaving the setup screen. On load, each card asynchronously syncs from the server if no local save is present, updating in-place when the data arrives. The last-played character's card is pre-selected on boot.

### World

### Done

- **US-031** As a player, I can open a conversation with an AI dungeon master (AIDM) at any time during an encounter so that I can interact with the world through free-form text. A persistent "DUNGEON MASTER" button in the HUD opens a full-screen chat overlay powered by Claude Sonnet. The AIDM receives complete, real-time encounter state on every message: player stats (HP, XP, gold, AC, inventory, hidden status, action/bonus used, moves left, Second Wind charges), all enemies (HP, AC, position, active-turn flag, vexed/hidden status), all NPCs (name, id, position), quest progress, items on the ground, secrets remaining, the selected target, and the last 20 combat log entries. The AIDM can control all aspects of the encounter via tool use: adjust player HP, award XP/gold, set enemy HP (killing enemies properly via the combat system), add narrative log entries, teleport entities (enemies resolved by label or array index; NPCs by their stable `defId`), give the player items, remove items from inventory (thrown javelins, consumed reagents, lost gear), start or end combat, force-complete quests, toggle player stealth, spawn a new enemy onto the map (placed on the nearest free tile and added to the combat initiative order), and despawn an NPC (removes the entity from the map with a narrative reason). The selected target is surfaced to the AIDM with its entity id both in the focus line and the NPC list. When `trigger_combat` is called in a social encounter with no enemies, all present NPCs are converted to enemy combatants in-place (same position and stats), cleared from the NPC roster, and combat begins. When moving multiple entities to the same tile, each is placed on the nearest free passable tile that is not already occupied by any other entity. The AIDM roleplays all NPCs in their own voice, constrained by their persona and knowledge. When the DM overlay is re-opened after the player has taken actions, new combat log entries since the last conversation are automatically injected as a context update so the DM is always current. Pronouns ("them", "it") are resolved to the player's currently selected target. Conversation history is preserved across open/close cycles for the duration of the encounter.
