## Functional Requirements and Roadmap

---

### Core Rules

All D&D 5e SRD mechanics: movement, combat, action economy, checks, saves, conditions, and damage rules.

#### Done

- **US-001** As a player, I want to move my character around the world using keyboard input (WASD and arrow keys) so that I can explore the game world. The world is represented as a tile-based map where each tile equals 5 feet, making movement structured and tile-aligned.
- **US-005** As a player, I want to engage in turn-based combat with enemies so that exploration feels challenging and rewarding. Combat is fully integrated into the map: movement (30 ft = 6 tiles per turn) and actions happen on the same map, following SRD 5.2.1 rules (initiative, attack rolls, weapon properties and masteries, class features, monster traits, death saving throws). A Critical Hit (natural 20) rolls all damage dice twice including Sneak Attack dice. Making a ranged attack (including thrown weapons) while within 5 feet of an enemy who can see the attacker and does not have the Incapacitated condition imposes Disadvantage on the attack roll. Death Saving Throws follow the full SRD rules: rolling a natural 1 counts as two failures; rolling a natural 20 immediately restores 1 HP and ends the Unconscious condition; taking any damage while at 0 HP causes one automatic failure (two if the damage is from a Critical Hit); accumulating 3 successes makes the character Stable, which stops further death saves without restoring HP. The event log is a two-column display: the left column shows immersive narrative ("Hit — 8 slashing"), the right column shows dice detail ("d20(14)+5=19 vs AC 14 · 1d6[5]+3"). Each row is colour-coded by outcome. It is possible to scroll through the entire event log.
- **US-017** As a developer, I want every creature's stat block to be sourced directly from the D&D 5e SRD 5.2.1 so that combat mechanics are correct and new creatures can be added without guesswork. Monsters (`MonsterDef`) hold SRD stat blocks: creature type, HP dice formula, full ability scores, an explicit `initiativeBonus` field, attack entries with attack type (melee / ranged / both), reach, and range, plus optional `resistances` / `vulnerabilities` / `immunities` / `conditionImmunities` arrays and `nimbleEscape` / similar trait flags. NPCs (`NPCDef`) are a separate type that references a `monsterClass` for their stat block while adding identity fields (name, colour, persona). The combat system selects the correct attack type based on range. New monsters drop into `server/data/monsters/` as JSON files; the count and identities are content, not requirements.
- **US-019** ~~Procedural map generation~~ Superseded by US-023 / US-064. Encounter Setup always uses `mapType: "saved"`; every encounter references a hand-authored Tiled map. The procedural `open` / `rooms` map types remain available in the server request schema but are no longer reachable from the UI. Diagonal-corner movement, flood-fill connectivity, and BFS pathfinding are still part of the engine and apply to all maps (saved or procedural).
- **US-024** As a player, I want to fight multiple enemies in a single combat encounter so that battles feel more strategic. Simple Combat spawns 2–4 enemies chosen randomly from the available monster pool in `server/data/monsters/`. All map enemies join combat the moment one triggers it. Enemies and allies share a single uppercase A–Z combat-label pool (`assignCombatLabel` picks the lowest unused letter regardless of disposition), so the first enemy and the first ally in the room might be labelled "(A)" and "(B)" respectively. The label is rendered in the centre of the token during combat and in the Turn Order Bar at the top of the Game Map. All non-neutral NPCs (enemies and allies) act sequentially during the NPC phase; enemies attack the player and allies attack the nearest enemy. The active chip in the Turn Order Bar is highlighted green and dead chips are dimmed. Combat ends only when all enemy-disposition NPCs are defeated; surviving allies revert to neutral disposition.
- **US-054** As a player, I want NPCs to have a disposition (ally, neutral, or enemy) so that the same entity can shift allegiance during play without losing its identity. All creatures — enemies, allies, and social NPCs — share a single `NpcState` type differentiated by `disposition: 'ally' | 'neutral' | 'enemy'`. Enemy-disposition NPCs attack the player in combat; ally-disposition NPCs join the initiative order and automatically attack the nearest enemy on their turn; neutral-disposition NPCs stand in the world and do not participate in combat. Rendering is disposition-aware: all three dispositions show an HP bar overlaid at the top of the token circle when damaged — red for enemies, green for allies, blue-grey for neutrals. All NPCs show their name (or revealed name) just above the token. Enemy NPCs receive a `combatLabel` (A, B, C…) at spawn; ally NPCs receive a `combatLabel` from the same pool when combat starts or when their disposition changes. `combatLabel` is rendered in the centre of the token during combat and hidden while exploring. Token disposition is synced from server state on every tick. When combat ends, surviving allies revert to neutral; living enemies are removed from the map. Dead NPCs (HP = 0) remain on the map as corpses regardless of disposition: their token is rendered at 40% opacity with no HP bar and no label. Corpses are selectable — clicking one opens the Target Panel showing their stats — but they cannot act, be targeted by attacks, or trigger combat. The AIGM receives dead NPCs in a separate CORPSES section of the game state and can direct the player to search them. The AIGM can change any NPC's disposition at any time via `set_disposition`. Each NPC also carries an optional `revealedName` field set by the `reveal_npc_name` AIGM tool; once set, it replaces the generic name displayed above the token and is used by the sayto mode button in the HUD GM tab. An optional `combatPassive` boolean flag (set via `set_npc_passive`) causes an ally to skip their combat turn entirely, allowing the player to instruct allies to stand down without changing their disposition.
- **US-055** As a player, I can attack a neutral NPC outside of combat to start a fight, so that I have agency over when and against whom combat begins. Selecting a living non-ally NPC and pressing ATTACK during the exploring phase promotes that NPC to enemy disposition, triggers faction aggro, and then starts combat via a normal initiative roll. Faction aggro: every NPC carries a `factionId` (defaults to its definition id, so all instances of the same NPC type share a faction); when any NPC turns hostile all neutral NPCs sharing that `factionId` immediately join as enemies before initiative is rolled. Faction aggro fires from both player-initiated attacks and AIGM `set_disposition` calls, ensuring the mechanic is consistent regardless of how hostility starts.
- **US-025** As a player, I want the action economy to follow D&D 5e SRD rules so that combat feels authentic. Each player turn grants one Action, one Bonus Action, and one free object interaction. Attack consumes the Action; Second Wind (Fighter) consumes the Bonus Action; Drinking a Health Potion costs the Bonus Action in combat and is free during exploration. **Hide** costs the Action by default, but a Rogue of level 2 or higher gets **Cunning Action** which makes Hide a Bonus Action. **Equipping or unequipping a weapon or shield** during `player_turn` uses the free object interaction the first time per turn; a second swap that turn requires the Utilize action and consumes the Action (server enforces; the engine refuses with an in-fiction log line when both are spent). **Equipping or unequipping armor** is blocked entirely during combat (SRD donning times are 1–10 minutes — impossible to fit into a 6-second round). The player remains in their turn after spending either resource and must press End Turn to hand initiative to the enemies. Action Buttons that require a spent resource are dimmed and disabled.
- **US-042** As a player, I want Dash, Dodge, and Disengage actions so that I have tactical options beyond attacking. **Dash**: spend Action to gain extra movement equal to Speed for the turn. **Dodge**: until start of next turn, attacks against the player have Disadvantage and the player has Advantage on DEX saves; shown in Phase Text. **Disengage**: movement does not provoke Opportunity Attacks for the rest of the turn. These appear as Action Buttons in the Player Panel when the Action has not yet been spent.
- **US-043** As a developer, I want a formal Advantage/Disadvantage system so that all D20 Test modifiers are consistent and composable. Any D20 Test can be tagged with Advantage or Disadvantage sources. When resolving the roll: Advantage → roll 2d20, take higher; Disadvantage → take lower; both present → cancel, roll normally. Sources include: player hidden (Advantage on attack), Vex mastery (Advantage on attacker's next attack roll against the same target), Dodge (Disadvantage on attacks against dodger), Surprise (Disadvantage on surprised creature's Initiative), ranged attack while adjacent to a conscious enemy (Disadvantage). The event log shows both dice values when the mechanic applies.
- **US-041** As a player, I want every creature — player, allies, enemies, and NPCs — to make Opportunity Attacks when an opponent moves out of their reach, so that positioning carries meaningful risk for everyone in the encounter. When a creature uses its movement to leave the reach of an opponent that can see it, that opponent may spend its Reaction to make one melee attack. **One Reaction per round per creature**, refreshed at the start of *that creature's own turn* (not at the start of the player's turn) — so an enemy that burned its Reaction during its turn cannot react again until it acts again. **Visibility gate**: a creature cannot take an Opportunity Attack against a target that is `hidden` or `invisible` (the SRD's "a creature you can see" clause; implemented via `ConditionSystem.isVisible`). **Incapacitation gate**: a creature with the Incapacitated, Stunned, Paralyzed, or Unconscious condition cannot take Reactions at all. The four trigger paths are all wired: (a) the player OA against an enemy that moves away — surfaced via the Reaction Prompt Overlay (see US-067), never auto-fires; (b) enemy OA against the player on player movement — auto-fires; (c) ally OA against an enemy on the enemy's turn — auto-fires; (d) enemy OA against an ally on the ally's turn — auto-fires. Disengage suppresses Opportunity Attacks against the player for the turn. The event log notes each Opportunity Attack with both the attacker and the target identified by their disambiguated combat label.
- **US-078** As an encounter author, I want to start a new encounter from an existing saved map without re-composing one, so I can build several encounters that reuse the same layout. The **PICK MAP** button on the Generator Setup Scene's Deterministic tab (bottom bar, left of COMPOSE MAP) opens the `MapSelectorOverlay` (`client/src/ui/generate/MapSelectorOverlay.ts`) — a modal grid of cards showing every saved map from the `maps` registry. Each card renders a small thumbnail using the map's own multi-tileset routing (so water + scribble + dungeon tiles all decode correctly), the map name, and the map description. Clicking a card converts the `SavedMapDef` into a `MapPreviewData` (mapId set, terrain/object data flattened, tileset sources reversed back to `../tilesets/<name>.tsj`), assigns it as the scene's `acceptedMap`, clears any pending RANDOMIZE seeds via `clearRolledState`, and rebuilds the right panel so the encounter builder unlocks without going through COMPOSE MAP. The bottom bar layout is now five-wide: BACK | ★ RANDOMIZE | PICK MAP | COMPOSE MAP | SAVE ENCOUNTER, with widths trimmed so the dev button (DELETE ALL GEN MAPS) still fits at the right without shifting any non-dev element.
- **US-077** As a player, I want the Adjudicator to roll a complete encounter on its own that I can then inspect, edit, and save, so authoring a fresh scenario is a one-click starting point — not a black box. The **★ RANDOMIZE** button on the Deterministic tab runs a data-driven roll: pick a weighted entry from `client/src/data/encounterArchetypes.ts` (the `ENCOUNTER_ARCHETYPES` registry), roll its content (features rolled from `featurePicks`, monster ids drawn from `enemyPool`/`allyPool` with replacement up to count ranges, title / introduction / objective / description picked from string pools), composes the map in memory (NOT persisted yet), then **populates the existing Adjudicator-tab form**: terrain + features chips flip to rolled values, title / intro / description / objective / completion-flag DOM inputs are filled in, the zone painter is seeded with PLAYER (blue) + NEUTRAL (amber) cells, the monster picker is seeded with rolled ally + neutral ids (rolled hostile-intent monsters go into the NEUTRAL bucket so the encounter starts in exploration phase), and the trigger editor is seeded with rolled triggers. **Spawn placement is anchor-driven** so it's suitable for the rolled terrain and story: `MapComposer` returns a `MapAnchors` payload (entrance + vault + rooms for dungeons, campfires for campsites, building / ruin footprints, path endpoints, inland band for coastline) alongside the grid; each archetype declares ordered `playerAnchors` and `enemyAnchors` (e.g. Dungeon Sweep uses `['entrance', 'edge:south']` for the player and `['vault', 'far_room', 'edge:north']` for enemies; Bandit Camp uses `['away_from:campfire', 'edge:south']` for the player and `['campfire', 'edge:north']` for enemies); the randomizer walks each list and paints cells around the first anchor present, with `edge:*` fallbacks guaranteeing placement when a feature placer didn't fire. **Triggers are rolled too**: each archetype carries `triggerTemplates` declaring 2-3 anchor-pinned triggers across the four trigger kinds (perception checks at the path / ruin / far room, atmospheric log entries near campfires, AIDM cues at the vault door, scripted combat starts at the path's far end / inside the bandit camp / inside the ruined nave / in the dungeon vault); `rollTriggersFromAnchors` resolves each template's anchor into a `{x,y,w,h}` region clamped to the rolled map and silently drops templates whose anchor didn't materialise. **Encounters start in exploration, not combat**: rolled monsters spawn with `disposition: 'neutral'` (rolled enemy ids are routed to `npcIds`, not `enemyIds`), so the session-create auto-combat check in `index.ts` no longer trips. Combat starts via either (a) a `combat` kind trigger that flips every rolled type to enemy and calls `trigger_combat`, or (b) the player attacking a neutral and faction aggro waking the rest. The combat trigger now accepts a `defIds: string[]` field alongside the existing single `defId`; the server unions both into a deduped flip list before `trigger_combat`. The randomizer populates `defIds` with every unique rolled enemy id so a single combat trigger flips a multi-type pool (e.g. Dungeon Sweep's `['skeleton', 'kobold_warrior']`). **Trigger regions are visualised** on both the right-panel thumbnail (`ZonePainter.setTriggerRegions`) and the large `MapPreviewOverlay` as colour-coded outlined rectangles (perception = teal, log = pale blue, AIDM = amber, combat = red) so the author sees exactly which tiles will fire each trigger; the thumbnail re-renders on every TriggerEditor edit via an `onChange` callback. **Map persistence is deferred**: the rolled map lives in memory until either (a) the user clicks the SAVE button in the COMPOSE MAP preview overlay, or (b) the user clicks SAVE ENCOUNTER, in which case `runComposeEncounter` saves the map on the fly before composing the encounter shell. Picking a different map via PICK MAP, or revisiting the scene via BACK + GENERATE ENCOUNTER, clears the pending roll seeds (including the in-memory map). The system is intentionally extensible: adding new content is a single new entry in `ENCOUNTER_ARCHETYPES` (or extending an existing entry's `triggerTemplates` / pools); adding a new anchor type is a `MapAnchors` field plus a case in the randomizer's `resolveOne` + `regionFromAnchor`. Placement and rolling logic live in the pure module `client/src/encounterRandomizer.ts` (`pickArchetype`, `rollArchetype`, `buildStartingZonesFromAnchors`, `rollTriggersFromAnchors`) so future server-side / AI-seeded variants can reuse them. The shipped roster covers five archetypes (Forest Ambush, Bandit Camp, Sunken Shore, Crossroads Skirmish, Dungeon Sweep) that exercise all three terrains, the anchor vocabulary, and all four trigger kinds.
- **US-076** As an encounter author, I want the deterministic engine's Event Log to be sufficient on its own so a player can finish an encounter without ever opening the GM tab. The HUD's left tab was renamed from **COMBAT LOG** to **EVENT LOG** and the underlying field from `state.combatLog` to `state.eventLog` (with `loadWorldState` migrating the old field name on read so resumed sessions don't crash). At session creation, `SessionBuilder` seeds the log with three deterministic entries: the encounter title as a `header`, each line of the introduction prose as `status` entries, and the `Objective:` line — so a player can scroll back and re-read scene context that the Introduction Overlay would otherwise only show once. The deterministic engine writes every mechanical event into the log via `ctx.addLog`: combat actions (attacks, hits, misses, damage, kills, dashes, dodges, disengages, opportunity attacks), item pickups, concentration changes, spell casts, class-feature uses, scripted-event narration variants (from `TriggerSystem.narrate`), and combat-start / turn-header / combat-end markers. Combined with the seed, the log is sufficient to play a full encounter without ever opening the GM tab — the GM panel is strictly additive (free-form roleplay, improvisation, and prose), and `DevMode.disableAIGM` (see US-068) makes that explicit by short-circuiting the GM with a canned silent reply.
- **US-075** As a player, I want the turn-order bar to show creature artwork so I can identify combatants at a glance, with the active creature visually emphasised. The turn-order strip pinned to the top of the Game Map (`HUD.refreshTurnOrder` in `client/src/ui/HUD.ts`) is **transparent and borderless** — chips float against the canvas with no panel chrome. Each chip is a square SVG token tile (the same artwork as the on-map token) at 36 × 36 px; the chip representing the **active combatant grows 30% to ~47 × 47 px** and extends downward from the shared top baseline so resting chips stay aligned. There are no boxes, borders, or fills around the tokens — the size bump is the active-state indicator. NPC chips carry a small bottom-right combat-label badge (`A`, `B` …) where applicable; the player chip carries no badge. Hovering a chip surfaces the creature's name via the browser tooltip. Because the server processes each NPC's turn atomically (sets `npc.isActive = true`, generates all events, resets it to false), the client also drives the highlight from animation tracking: while an `entity_move` event for an NPC plays, that NPC's chip reads as active even though the server has already advanced past their turn. Without this client-side override the chip would never light up for an NPC mid-turn. The `animatingEntityId` field in `GameScene` holds the currently-animating entity; `buildHUDState` ORs it into each chip's `isActive` calculation.
- **US-074** As a player, I want creatures rendered with hand-drawn SVG tokens instead of coloured circles so the map and turn-order bar are more visually distinct. Every `PlayerDef`, `MonsterDef`, and `NPCDef` resolves to an SVG token via the helpers in `client/src/data/tokens.ts`. The naming convention is `/tokens/player_<className_lower>_<lineage|species, dashes stripped>.svg`, `/tokens/monster_<id>.svg`, and `/tokens/npc_<id>.svg`. The defs' optional `tokenAsset` field overrides the convention when the artist wants a non-conventional name (e.g. `wanderer` → `/tokens/npc_wandering_sage.svg`). For NPCs whose convention path doesn't exist on disk, the engine falls back to their `monsterClass`'s token at render time via `GameScene.resolveMonsterDef`. Files are served by the static route `GET /tokens/:filename` (whitelisted to `^[A-Za-z0-9_-]+\.svg$`, `Content-Type: image/svg+xml`); a sibling `GET /tokens` returns the directory listing so `BootScene.preload` only queues convention paths whose files actually exist — silent fallback for NPCs without dedicated art, no boot-time load errors. `BootScene` rasterises each SVG at 2 × tile size (100 × 100 px) for retina sharpness; `Player.ts` and `NpcToken.ts` render the resulting texture via `scene.add.image(...).setDisplaySize(TILE_SIZE − 6, ...)`. If a texture is missing at render time, the body falls back to a coloured circle in `DEFAULT_TOKEN_COLOR` (`0x3388ff`, exposed from `client/src/constants.ts` alongside `DEFAULT_TOKEN_COLOR_HEX` (`'#3388ff'`) — also the unified nameplate text colour). The same SVG resolver drives the character-card avatar on `EncounterSetupScene` / `AdventureSetupScene` and the turn-order chip art (see US-075). A **LABELS toggle chip** in the GM panel's chip row hides / shows NPC nameplates globally (default off so the map opens with a clean view); newly-spawned NPCs honour the current preference via `NpcToken.setNameVisible`.
- **US-073** As a developer, I want the NPC `hold` behaviour to take the SRD Dodge action instead of standing inert, so a creature without a useful attack still gets a defensive benefit. When `chooseNpcBehavior` returns `'hold'` and the NPC is not Incapacitated, `runSingleEnemyTurn` and `runSingleAllyTurn` push `'dodging'` onto the NPC's conditions and log `"<name> Dodges — attackers have Disadvantage"` instead of the inert `"<name> holds its ground"`. The dodging condition is consulted by the attack resolver (`target.conditions.includes('dodging')` → attacker rolls with Disadvantage). To make Dodge actually protect the NPC against the player's next swing, **turn-scoped conditions (`dodging`, `disengaged`, `dashing`, `slowed`) now clear at the start of THAT creature's own next turn** rather than at the start of the player's turn (which was the previous behaviour — Dodge / Disengage from an NPC was wiped *before* the player's attacks resolved, making them functionally useless). The clear happens inside `runSingleEnemyTurn` / `runSingleAllyTurn` alongside the reaction reset; the equivalent clear for the player still fires in `enterPlayerTurn`. Incapacitated creatures fall back to the bare hold message since they can't take an action.
- **US-072** As a player, I want to iterate on a map design without committing to a full encounter, so I can browse layouts until I see one I like. `GenerateSetupScene` is structured as **two tabs** — **DETERMINISTIC** and **GENERATIVE AI** — with no character selector on this screen; once an encounter is authored the player is handed off to `EncounterSetupScene` with the new encounter pre-selected (via `init({ presetEncounterId })`), and the character pick happens there. The Generative AI tab's left panel is a free-text prompt textarea plus encounter-type chips; the right panel is a vertical stack of clickable **example prompt cards** (Moonlit Graveyard, Goblin Warren, Riverside Ambush, Abandoned Watchtower, Crossroads Market, Wolf Den) — clicking a card copies its body into the textarea for editing. Its buttons are **GENERATE MAP ONLY** (`POST /generate/map` → `encounterGenerator.generateMap`, returns just `{ name, description, width, height, terrainData, objectData }`) and **GENERATE ENCOUNTER** (`POST /generate/encounter` with the prompt + selected types, then navigates to EncounterSetupScene with the new encounter id). The Deterministic tab's left panel exposes the MapComposer toggles (TERRAIN: Grassland / Forest; FEATURES: Ruins / Buildings / Campsites / Path); its right panel exposes encounter-level settings (encounter-type chips + a DESCRIPTION textarea that becomes the new encounter's `customContext` so the in-game GM has scene context to work from). The deterministic right panel is **gated on an accepted map**: until the user presses ACCEPT in the Map Preview Overlay, the right side only shows the label `No map available` with no other controls. Once a map has been accepted, the right side exposes a thumbnail of the accepted map (also acting as a click-and-paint surface for **starting zones** — PAINT: PLAYER paints blue cells, PAINT: ENEMY paints red cells, click-and-drag paints continuously; clicking the thumbnail while no paint mode is active instead opens the Map Preview Overlay in view-only mode with the painted zones drawn on top so the player can inspect the map at higher zoom), encounter-type chips, a description textarea (becomes the encounter's `customContext`), and a **monsters picker** — a scrollable list of every entry in the `monsters` registry where each row carries `+ ALLY` and `+ ENEMY` buttons to add that monster to the encounter's `allyIds` (friendly disposition) or new `enemyIds` field (hostile, with auto-assigned combat labels). Both ally and enemy ids are resolved against `npcs/` first and then `monsters/`, so raw monster defs (`guard`, `bandit`, `wolf`) work without authoring an NPC wrapper. The engine **auto-starts combat** at session creation whenever any NPC has `disposition: 'enemy'` and live HP — so a deterministic-compose encounter with enemyIds lands the player directly in the initiative tracker (with a `⚔ Combat begins` log entry) as soon as the introduction overlay is dismissed; no GM `trigger_combat` call is needed for the obvious case. Re-opening the preview and accepting a different map resets all painted zones + monster selections. Its buttons are **COMPOSE MAP** (preview-only via `POST /generate/map/composed` → `MapComposer.composeMap`) and **COMPOSE ENCOUNTER** (`POST /generate/encounter/composed` — passes `existingMapId` for the accepted map, the painted `startingZonesData` array, the selected encounter types, the description, and the expanded ally + enemy id lists; the server writes the encounter JSON referencing the existing map, with no second composing step). All buttons in the generator scene follow a **stay-clickable-when-disabled** pattern: instead of silently rejecting clicks, a greyed button surfaces a specific hint in the status line explaining the missing precondition (e.g. "Compose a map and press ACCEPT in the preview first." or "Paint at least one player-start cell on the thumbnail (PAINT: PLAYER)." for COMPOSE ENCOUNTER). Both paths produce `gen_<timestamp>_<slug>` files that are picked up by `loadDefs()`. The composer's core support primitive is `stampRoom(terrain, opts)` which lays down a rectangular room with correctly-rotated walls (`stone_wall_top` rotated for N/S/E/W edges), corner tiles (`stone_wall_corner_tl` rotated for the four corners), an interior of floor with an optional accent-tile checker (`stone_floor` + `stone_floor_cracked` mix), and any number of carved-out doorways; ruined rooms additionally take a `ruinedBreaks` count that knocks out random non-corner wall cells via a seeded Fisher-Yates pick. `composeMap()` layers terrain (grass everywhere, plus ~22% tree density on the object layer for Forest with a sparser south belt for player spawn space), then stamps the chosen features on top (path is laid first as a meandering N↔S or E↔W dirt trail using `path_straight_v` ± rotation, so subsequent buildings / ruins / campsites can stamp over it where they overlap). Map seeds default to `Date.now()` so REGENERATE produces a fresh layout. The shared `MapPreviewOverlay` (`client/src/ui/MapPreviewOverlay.ts`) renders the actual tileset sprites at 14 px per tile inside a clipped 1020×440 viewport; the user can **zoom** with the mouse wheel (centred around the cursor, clamped to 0.5×–4×) and **pan** by click-and-dragging within the viewport. The overlay's button row adapts to which callbacks the caller supplied: COMPOSE MAP wires `onRegenerate` + `onAccept` for the full editor (REGENERATE / ACCEPT / CLOSE); GENERATE MAP ONLY wires `onRegenerate` only (REGENERATE / CLOSE); the click-to-enlarge from the thumbnail wires neither and renders just a centred CLOSE button. An optional `zones: { playerCells, enemyCells }` parameter draws blue / red 50%-alpha overlays on top of the matching grid cells, living inside the grid container so they zoom and pan with the map. **Dev-mode "Delete all generated maps"** is a corner button on `GenerateSetupScene` gated behind `DevMode.enabled` — it calls `DELETE /generate/maps/all` which unlinks every `gen_*.json` from `server/data/maps/` and `server/data/encounters/`, then re-runs `loadDefs()`. The button is slotted into leftover space at the bottom-right and doesn't shift any non-dev layout (see [CLAUDE.md](CLAUDE.md) "Dev Mode Buttons").
- **US-071** As a player, I want to **generate** a one-off encounter from a free-text scene description so that I can play a brand-new scenario without authoring a JSON file. A new top-level menu option (Boot → MainMenuScene → GENERATE ENCOUNTER → `GenerateSetupScene`) opens the tabbed generator screen. On the **GENERATIVE AI** tab the player picks zero or more encounter-type chips (combat / social / exploration) and either types a free-text scene description into the prompt textarea or clicks one of the example cards on the right panel to seed it. `POST /generate/encounter` calls Claude Sonnet with a structured `submit_scenario` tool whose input schema covers the full map + EncounterDef payload. The system prompt grounds the model with the live **tile legend** (every GID, name, passability, layer, description from `server/data/tilesets/*_legend.json`), the **monster roster**, and the **NPC roster** so it can only emit ids that actually exist. `server/src/encounterGenerator.ts` then validates the output (every terrain GID present in the legend; object-layer GIDs valid; data arrays sized to width × height; at least one player-start tile in `startingZones`; referenced npcIds / allyIds exist; non-combat encounters declare a `completionFlag`) before writing two files: `server/data/maps/gen_<timestamp>_<slug>.json` and `server/data/encounters/gen_<timestamp>_<slug>.json`. Both files set `generated: true` so the Encounter Setup card renders a `✦ GENERATED` badge — they're full first-class encounters that can be replayed indefinitely. After writing, the server re-runs `loadDefs()` so the new map joins `defs.maps`. The client then navigates to `EncounterSetupScene` with `init({ presetEncounterId: <new id> })`; that scene refreshes its encounter list from the server if the new id isn't cached yet and pre-selects the new encounter so the player only has to pick a character and press BEGIN. Malformed model output is rejected with a precise error string (e.g. "terrainData[37] references unknown GID 999"); the player can refine their prompt and retry without server restart.
- **US-070** As a player, I want to play **Adventures** — a string of encounters with overarching narrative and choices that carry across chapters — so that what I do in chapter 1 shapes what happens in chapter 3. Chunks written before a roll-requesting tool resolves are speculative; the server rolls them back via `aigm_speculative_discard` and the real text follows after the roll. Chunks accompanying non-speculative tools are confirmed with `aigm_checkpoint`. A final `aigm_done` carries the canonical reply and any roll-result strings (rendered as 🎲 entries). The AIGM receives complete, real-time encounter state on every message: player stats (HP, XP, gold, AC, inventory, hidden status, action/bonus used, moves left, class-feature resources, spell slots, concentration target), all living combatants (enemies and allies: HP, AC, position, active-turn flag, vexed/hidden status, disposition, attacks, revealed name if any), all living neutral NPCs (defId, entity ref, position, revealed name if any), dead NPCs as a separate CORPSES section (name, position — searchable but cannot act), quest progress, items on the ground, secrets remaining, the selected target, and the full event log. The AIGM can control all aspects of the encounter via tool use: adjust any entity's HP (`adjust_npc_hp`: entity format `"player"`, `"enemy_A"`, or `"npc_[id]"`; positive delta heals, negative damages; kills the entity when HP reaches 0), award XP/gold, add narrative log entries, teleport entities (`move_entity`), give the player items, remove items from inventory, start or end combat, force-complete quests, toggle player stealth, spawn a new enemy onto the map, despawn an NPC, change any NPC's disposition (`set_disposition`), apply or remove conditions, throw any inventory or map item at an enemy (`throw_item`: proper thrown weapons use weapon stats; other items are improvised 1d4 bludgeoning), reveal an NPC's name (`reveal_npc_name`: sets `NpcState.revealedName` and updates the token name label; must be called before narrating the name so the UI reflects the reveal), mark an ally as combat-passive (`set_npc_passive`: when `passive: true` the ally skips their combat turn entirely; reversed with `passive: false`; used when the player tells an ally to stand down or stay back), and search the full conversation archive for older content (`recall_memory`: case-insensitive substring search, returns up to 8 newest-first matching snippets — used when the sliding-window history has been summarised and the model needs to look up an earlier exchange), durably shift the player's reputation with a faction (`adjust_faction_standing`: writes `GameState.factionStandings`, clamped to [−100, +100], surfaced to the GM in CURRENT STATE under FACTION STANDINGS), and record a significant world event into long-term world memory (`create_rumor`: appends to `GameState.rumors` idempotently by id, surfaced to the GM under the RUMORS block — see US-069). The selected target is surfaced to the AIGM with its entity ref in the focus line. When `trigger_combat` is called in a social encounter with no enemies, all present NPCs are converted to enemy combatants in-place. When moving multiple entities to the same tile, each is placed on the nearest free passable tile. The AIGM roleplays all NPCs in their own voice, constrained by their persona and knowledge. Pronouns ("them", "it") are resolved to the player's currently selected target. Conversation history is stored server-side in the session and preserved for the duration of the encounter; the client retains a local copy for display. To keep cost and latency bounded the server maintains two histories per session: a **sliding window** (`aigmHistory`) of the most recent ~20 messages that is what's actually sent to Claude on each turn, and a **full archive** (`aigmArchive`) of every exchange that's only consulted by the `recall_memory` tool. When the sliding window exceeds 40 messages the oldest entries are collapsed into a single `[SUMMARY OF EARLIER TURNS]` assistant message generated by Claude Haiku, and the most recent 20 messages are kept verbatim. The full archive is in-memory only and resets if the server restarts. The system prompt is split into a static instructions section (eligible for Anthropic prompt caching) and a dynamic state section injected into each user message. Historical `[CURRENT STATE]` blocks are stripped from the conversation history before each API call so the model always reasons from the freshly injected state. Both the system prompt and the tool list carry `cache_control: ephemeral` markers, and inside the tool-use loop the most-recent `tool_result` block also gets a cache_control marker so long tool chains don't re-pay the prefix. The loop is capped at 8 iterations: on the final iteration every tool result is overridden with a `TOOL BUDGET EXHAUSTED` signal and a tool-less follow-up forces the model to finalize its narrative. Each Claude API call is wrapped in a single retry-with-backoff on transient errors (408, 425, 429, 5xx, 529). Concurrent AIGM requests on the same session are rejected with HTTP 429 via a per-session mutex. The GM operates under these constraints: **TOOL-FIRST RULE** (every game effect must be enacted via the corresponding tool before it is narrated; `request_attack_roll` does not apply damage — the GM must follow up with `adjust_npc_hp`; `set_disposition` does not start combat on its own — the GM must call `trigger_combat` after disposition changes in the exploring phase; NPC name reveals require `reveal_npc_name` before the name appears in narration); **NPC NAMES** (when any NPC gives their name, `reveal_npc_name` must be called first with the entity ref from CURRENT STATE and the chosen name); **WORLD GROUNDING** (only references creatures, items, and events present in state or established in conversation); **TURN ORDER** (during `player_turn`, does not narrate or simulate enemy turns); **PLAYER AGENCY** (enacts any mechanically possible action if the player confirms after one warning); **STATE IS AUTHORITATIVE** (trusts current state over conversation history without comment); **SEARCHING CORPSES** (`request_ability_check` before narrating what is found). Gold transactions require `award_gold` before narration. Skill-based player actions require `request_ability_check`. The CURRENT STATE block also surfaces queued **SCRIPTED EVENTS** (lines pushed by `send_aigm_message` trigger actions — incorporate into the next reply, then the server clears them) and, when set, **FACTION STANDINGS** and **RUMORS** blocks for long-term world memory. The GM operates **out of the action critical path**: `POST /game/session/:id/action` resolves every player action through the deterministic engine without ever calling Claude, and the client's `DevMode.disableAIGM` flag (URL param `?disableAIGM=true` or localStorage) short-circuits the GM Chat callback so the encounter is fully playable on the deterministic layer alone (see US-068).

Armor AC is computed per SRD category (light: base+DEX; medium: base+min(DEX,2); heavy: fixed base) plus +1 for Fighting Style: Defense and +2 for a shield. Finesse weapons use the higher of STR/DEX for attack and damage. Character stats that depend on data objects are never stored in character JSON files — they are always computed at startup and session creation by three functions that run in order: (1) `applySpecies` reads `speciesId` and `speciesLineage` to derive `speed` (base species speed + any lineage bonus); (2) `applyFeats` reads `featIds` to derive `savageAttacker` and `fightingStyleDefense`; (3) `applyEquipment` reads `defaultEquipment` to derive `ac` and `mainAttack`. The Character Sheet Overlay's Inventory tab is opened from the CHARACTER button in the Player Panel — it shows the three equipped slots with computed stats and UNEQUIP buttons, a scrollable carried-items list with EQUIP buttons (identical items grouped with a ×N count), and a live AC + attack summary bar. Equipping a two-handed weapon auto-unequips any held shield. Equipping one item from a stack removes only that instance. Equipment changes broadcast to the client via `PlayerState.ac` so the displayed AC updates immediately. Equipped items are persisted in the save file. The AIGM is aware of equipped armor, weapon, and shield and can grant or remove items via `add_item` and `remove_item`.
- **US-033** As a player, I want to see my save information inside the character selection card so that I know where each character left off. Each character card on the **Encounter Setup Scene** and the **Adventure Setup Scene** shows HP / XP / GP from the saved state and the currently equipped weapon and armor. A DELETE SAVE button inside the card clears the save from localStorage and the server without leaving the setup screen — on the Adventure Setup Scene it also clears the adventure save for that character (so the player can replay the adventure from chapter 1 with default gear). On load, each card always syncs from the server: when the server returns a save it overwrites the local mirror and refreshes HP/XP/GP, equipped gear, encounter log, and story log in place; when the server returns `null` (no save file on disk) the local mirror is purged and the card resets to a "no save" state — DELETE SAVE and STORY LOG are dimmed and disabled. The `GET /save/:characterId` route uses `readSaveIfExists` so missing saves return `null` rather than a fabricated default, preventing stale localStorage from making freshly-deleted saves look active. The last-played character's card is pre-selected on boot.
- **US-059** As a player, I can read a Story Log which gives me an AI-generated prose recap of every encounter I have completed, so that I can build a narrative record of my character's adventures. A STORY LOG button on the character card in the Encounter Setup Scene opens the Story Log Overlay. The overlay lists all completed encounters in chronological order; each entry shows the date, encounter type, and a short prose narrative generated by Claude Sonnet. The narrative is written in third-person, two to three sentences long (one short paragraph), weaves the encounter description naturally into the story, translates mechanical events into fiction (hits become blows landed, kills become deaths), includes meaningful dialogue in italics, and omits all mechanical numbers. Narratives are generated on demand via `POST /save/:characterId/storylog`, which processes only encounters not yet in the log and appends new entries — old entries are never rewritten. A GENERATE N ENTRIES button in the overlay triggers generation and is disabled when the log is already up to date. A REWRITE ALL debug button regenerates all entries from scratch. Generated narratives are stored in the character save under `storylog` and persist across sessions. The generation logic lives in `server/src/storylog.ts`.
